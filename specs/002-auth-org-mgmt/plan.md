# Implementation Plan: Auth & Org Management

**Branch**: `feat/auth` | **Date**: 2026-06-16 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-auth-org-mgmt/spec.md`

---

## Summary

Implement email+password authentication, org creation, JWT session management, role-based access control, and team invitations using NextAuth v4 (CredentialsProvider) + bcryptjs against the multi-provider Postgres DB abstraction. This is module 001 and blocks all other modules.

---

## Technical Context

**Language/Version**: TypeScript 5 (strict)

**Primary Dependencies**:
- `next-auth` v4 — CredentialsProvider + JWT session strategy
- `bcryptjs` — password hashing at cost factor 12
- `resend` — transactional email for invite links
- `zod` — input validation on all Server Actions
- `react-hook-form` + `@hookform/resolvers` — form handling on signup/login/invite pages

**Storage**: PostgreSQL via `src/lib/db/index.ts` (provider-agnostic). Tables: `orgs`, `users`, `invitations`.

**Testing**: Vitest (unit) + Playwright (E2E)

**Target Platform**: Next.js 14 App Router on Vercel / local Docker

**Performance Goals**: Login completes in < 2s; signup in < 3s including bcrypt hash

**Constraints**: bcrypt max password length = 72 chars. Stateless JWT — role changes take effect on next session refresh, not immediately.

**Scale/Scope**: Single org per user (v1). Fixed roles: admin / member / viewer.

## Phase 0: Research Findings Summary

See [research.md](research.md) for full decisions. Key choices:

1. **NextAuth v4** (not Supabase Auth) — keeps auth provider-agnostic, consistent with reference implementation at `C:\Works\anticlock`
2. **bcryptjs cost 12** — standard, ~300ms, no native bindings needed
3. **JWT session strategy** — stateless, works on Vercel serverless
4. **`withAuth` middleware** — single file protects all routes
5. **Atomic transaction** for signup — org + user never partially created
6. **Invitations table** — new feature not in reference project; token = `crypto.randomUUID()`

## Phase 1: Design Decisions

### Auth pages UI (no Stitch screen)
Per Constitution Principle VIII, auth pages use shadcn Card centered layout:
- Wrapper: `max-w-md mx-auto mt-24`
- shadcn `Card > CardHeader + CardContent`
- Form fields: shadcn `Input` + `Label` + `Button`
- Error display: shadcn `Alert` with destructive variant

### getAuthUser() — used everywhere
Every Server Action across ALL modules imports and calls this as its first statement:
```typescript
const user = await getAuthUser()
if (!user) return { error: 'Unauthorized' }
```

### Invite flow
1. Admin submits email + role → `inviteUser()` creates `invitations` row + sends Resend email
2. Email contains: `{NEXTAUTH_URL}/invite/{token}`
3. Invitee opens link → `/invite/[token]/page.tsx` validates token, shows accept form
4. Invitee submits name + password → `acceptInvite()` creates user, marks invitation accepted
5. Client redirects to `/login` — invitee logs in normally

### Role change + JWT trade-off
Stateless JWTs cannot be immediately revoked. Role changes take effect on next login or session refresh (NextAuth default: 30-day JWT lifetime). Acceptable for v1.
